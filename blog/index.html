What Running LLM Inference in Production Actually Teaches You

Hard-won lessons from operating distributed LLM inference at scale with llm-d

MR
Markell Rawls AI Evangelist, Red Hat
June 2026 ~12 min read

Production LLM Inference is a Different Animal

If you have spent time training large language models, you know the drill: provision a cluster, launch a distributed job, wait hours or days, evaluate the result. Training is embarrassingly parallel. Inference is embarrassingly stateful.

The moment you deploy a model behind an API, you are in the critical path of user-facing applications. Latency budgets shrink from hours to milliseconds. Every incoming request is different: variable input lengths, unpredictable output lengths, wildly varying computational cost. Your system has to make real-time scheduling decisions thousands of times per second, and each decision carries direct user impact.

This is not "deploy the model and walk away." Serving LLM inference in production is more like running a high-throughput database than running a batch job.

The mental model shift is real. Inference serving shares more DNA with database query optimization than with training orchestration. You are managing scarce resources (GPU memory, compute bandwidth, network), handling concurrent access to shared state (the KV cache), making admission control decisions under load, and trying to maintain tail-latency SLOs while the workload pattern shifts constantly.

This post collects the lessons we have learned running llm-d in production since Red Hat AI 3 GA in October 2025. These are the things that surprised us, the patterns that saved us, and the observability practices that let us sleep at night.

The Top 5 Things That Surprise Teams Going to Production

Every team that moves from "it works on my GPU" to "it runs in production" hits the same walls. These are the five surprises that come up most often, regardless of infrastructure experience.

#1

Model loading is your longest deployment operation

A 70B parameter model is 140GB or more of weights. Loading those weights from remote storage over a network takes 4 to 15 minutes, depending on your storage backend and network topology. For teams accustomed to deploying web services in seconds, this is a shock.

The consequences ripple outward. Rolling updates become extremely expensive. If each pod needs 10 minutes to become ready, a fleet rollout that touches every replica can mean 30 minutes or more of degraded capacity. Scaling up in response to a traffic spike is too slow to matter -- the spike will have passed before your new pods are serving.

Solution Pattern

Node-local model caching, pre-pull DaemonSets that keep model weights warm on disk, and staggered rollouts with maxSurge=1 for GPU workloads. Treat model loading as infrastructure provisioning, not application deployment.

#2

KV cache is your most important (and fragile) resource

The KV cache holds the "memory" of every active conversation. It is stored in GPU HBM -- the most expensive memory in your entire cluster. Managing it well is the difference between a system that performs and one that collapses under load.

Prefix caching is one of the most powerful optimizations available. When many requests share the same system prompt (which is extremely common in production), the KV cache for that prefix can be computed once and reused, saving 50-80% of prefill compute. But there is a trap: cache eviction under memory pressure can cascade. Once you start evicting cached prefixes, every request that would have been a cache hit now needs full prefill compute, which increases GPU load, which causes more eviction, which causes more misses. This feedback loop can take a stable system to full performance collapse in seconds.

Solution Pattern

Frequency-weighted eviction policies that protect hot prefixes, cache pressure metrics that trigger alerts before the cascade begins, and admission control that sheds new requests at the edge rather than letting them compete for scarce cache space.

#3

Not all tokens are created equal

A request with 128K tokens of context is roughly 256 times more expensive to prefill than a request with 500 tokens. But both arrive through the same API, both enter the same queue, and to a naive round-robin scheduler, they look identical.

Without priority-aware scheduling, one massive-context request can starve hundreds of smaller requests. The expensive request monopolizes a GPU for seconds while short requests queue up, their TTFT climbing well past SLO thresholds. Users doing simple chat completions pay the latency price for someone else's 128K-context RAG query.

Solution Pattern

Context-length-aware priority classes, separate queues for different request tiers, and preemption support so that long-running prefills can be paused to service latency-sensitive short requests.

#4

GPU memory fragments just like disk

After hours of serving mixed workloads with varying context lengths, GPU memory becomes fragmented. You may have 8GB of free memory according to nvidia-smi, but you cannot allocate a contiguous block large enough for a 32K-context KV cache. Free memory does not equal allocatable memory.

This is one of the most pernicious production issues because standard monitoring does not reveal it. Your dashboards show plenty of headroom while users report request failures. The gap between "free" and "usable" widens over time, and without explicit defragmentation, the only fix is restarting the process.

Solution Pattern

Fragmentation ratio metrics that track the gap between total free memory and largest allocatable block. Alerts when the ratio exceeds 30%. Periodic defragmentation cycles during low-traffic windows. Graceful pod recycling on a schedule to reset memory state before fragmentation becomes critical.

#5

"Healthy" does not mean "ready"

Kubernetes health checks confirm that your pod process is running and responsive. But there is a dangerous gap between "process is alive" and "model is loaded and ready to serve." The standard /health endpoint returns 200 before model weights are fully loaded onto the GPU. During this window, traffic may route to pods that accept the connection but cannot actually serve inference.

In a disaggregated serving topology, the problem gets worse. The pod might be healthy, the model might be loaded, but the RDMA network path to the decode pool might be partitioned. The health check says everything is fine while the pod silently drops or fails every request that requires cross-pool communication.

Solution Pattern

Custom readiness probes that verify the model is fully loaded on GPU, warmup inference completes successfully, and (for disaggregated deployments) network paths to all required pools are active. Do not mark a pod as ready until it can actually serve the workload it will receive.

Patterns That Prevent the Most Common Failures

Each of these patterns addresses a class of production incidents we have either experienced or observed across teams running LLM inference. They are not theoretical -- they are operational necessities.

Pattern 1

Treat Model Loading as Infrastructure

Model weights should already be on the node before your deployment starts. Use init containers to verify model availability and file integrity. Set maxSurge=1 for GPU workloads so you never restart more pods than you can afford to lose. Configure PodDisruptionBudgets to maintain minimum serving capacity during voluntary disruptions.

# Staggered rollout for GPU workloads
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0

The goal is to make model availability a precondition of deployment, not a side effect. Pre-pull DaemonSets can keep weights warm on local NVMe, eliminating the network transfer from the critical path entirely.

Pattern 2

Cache-Aware Everything

Route requests to pods that already have relevant prefix cache entries. This is one of the highest-leverage optimizations in production inference. A cache hit on a common system prompt can reduce TTFT from 800ms to under 100ms.

Monitor cache hit rate as a primary SLO metric, not a secondary diagnostic. When cache hit rate drops, investigate immediately -- it often predicts a broader performance degradation. Implement backpressure when cache pressure is high: it is better to briefly queue requests than to evict cached prefixes that will be needed again in milliseconds.

This is exactly what llm-d's Endpoint Picker Policy (EPP) does. It makes routing decisions based on real-time KV cache state, sending requests to the pod most likely to have a cache hit for that request's prefix.

Pattern 3

Separate Concerns with Disaggregated Serving

Prefill and decode have fundamentally different resource profiles. Prefill is compute-bound: it processes all input tokens in parallel and benefits from raw FLOPS. Decode is memory-bandwidth-bound: it generates one token at a time and is limited by how fast you can read KV cache from HBM.

Running both on the same hardware means neither workload gets what it needs. Disaggregated serving separates them into different pod pools, each optimized for its workload and scaled independently based on its specific bottleneck.

# Prefill pods: compute-optimized, fewer but larger GPUs
# Decode pods: memory-optimized, more HBM per GPU
# Scale each pool based on its bottleneck metric:
#   prefill  -> GPU compute utilization
#   decode   -> HBM bandwidth utilization

This is a core architectural pattern in llm-d, and it becomes increasingly important as context lengths grow.

Pattern 4

Graceful Degradation Over Hard Failure

When load exceeds capacity, the system should admit fewer requests rather than failing all of them. Shed load at the edge -- at the load balancer or gateway -- not at the GPU where a failed request has already consumed expensive resources.

Implement priority tiers so that critical traffic survives pressure that would overwhelm the full request mix. A production system should always be able to serve its highest-priority traffic, even during an incident.

Monitor finish_reason across all responses. A spike in finish_reason=length means the model is silently truncating responses because it is running out of allocated generation budget. Users see a response that just stops mid-sentence. This is a silent failure mode that no health check will catch.

Pattern 5

Configuration as a Rolling Update

Configuration changes to a distributed inference system are not instant. When you update a scoring weight in the EPP, or change a cache eviction threshold, that change needs to propagate across all replicas. During propagation, different replicas are running different configurations.

Treat configuration changes like code deployments. Version your configs. Use rolling propagation that coordinates across replicas. Never assume instant consistency in a distributed system -- the assumption that "I changed the config, so all pods have the new config" has caused more incidents than anyone would like to admit.

Monitoring and Observability for LLM Inference

Traditional web service monitoring does not cover what matters in inference serving. GPU utilization alone tells you almost nothing useful. Here is what actually matters, what to alert on, and what to put on your dashboards.

The Metrics That Actually Matter

TTFT Time to First Token. The user-facing latency metric. This is what users perceive as "how fast the model responds." Track p50, p95, and p99.
ITL Inter-Token Latency. How fast tokens stream after the first one. Directly impacts the perceived "typing speed" of the model for streaming responses.
Cache Hit Rate Your efficiency indicator. A high cache hit rate means the system is reusing prefill compute. A dropping cache hit rate predicts imminent latency degradation.
Queue Depth Per-pod request queue depth. Your saturation indicator. When queues grow, latency follows. Uneven queue depths across pods indicate a routing problem.
Frag Ratio The gap between total free GPU memory and the largest allocatable contiguous block. Your hidden time bomb. Rising fragmentation means eventual allocation failures.
finish_reason Distribution of response completion reasons. Your silent error detector. A spike in length means truncation; invisible unless you track it.
Model Load Time Time from pod start to model ready on GPU. Your deployment speed indicator. Track this to measure the effectiveness of model caching strategies.

The Alerts You Need Day One

TTFT p99 exceeds SLO threshold -- Your users are waiting too long. Investigate routing, cache state, and pod health.
Cache hit rate drops more than 20% below baseline -- Prefix cache is being evicted. Check memory pressure and admission rates.
Any pod with queue depth exceeding 2x the fleet average -- Hot-spotting. The router is sending too much traffic to one pod, or that pod is slow.
Fragmentation ratio exceeds 30% -- GPU memory is fragmenting. Schedule a defrag or graceful pod recycle before allocation failures begin.
finish_reason=length exceeds 5% of responses -- Silent truncation. Users are getting cut-off responses. Check generation budgets and context limits.
RDMA connectivity check failures (disaggregated serving) -- Cross-pool communication is down. Prefill and decode pods cannot coordinate.

The Dashboards That Save You at 3 AM

Request Flow Trace a request from load balancer through the EPP to the serving pod and back. Show latency at each hop. When something is slow, this dashboard tells you where.
Per-Pod Health GPU utilization, memory breakdown (total, used, fragmented), queue depth, and cache state for every pod. The per-pod view that lets you spot the outlier.
Fleet View Which pods are serving, which are loading, which are draining. The operational view that tells you how much capacity you actually have right now.
Trend Analysis Are metrics degrading slowly over hours or days? Catch gradual fragmentation, cache hit rate decay, and queue depth creep before they become incidents.

How llm-d's EPP Prevents Production Issues

The Endpoint Picker Policy is the control plane for inference routing in llm-d. Rather than distributing requests blindly with round-robin or random selection, the EPP observes the real-time state of every serving pod and makes informed routing decisions.

It continuously monitors queue depth, KV cache contents, GPU memory pressure, and model loaded status across the entire serving fleet. Every routing decision factors in this state, which is why many of the incidents described earlier in this post are preventable with EPP-informed routing.

  • Cache-aware routing -- Routes requests to pods that already have relevant prefix cache entries. When a request arrives with a system prompt that matches cached prefixes on a specific pod, the EPP sends it there. This can reduce TTFT from hundreds of milliseconds to tens of milliseconds for cache-hit requests.
  • Load-aware routing -- Avoids sending requests to pods that are already saturated. Instead of blindly distributing load, the EPP factors in each pod's current queue depth and GPU utilization, preventing hot-spotting before it causes latency spikes.
  • Health-aware routing -- Automatically routes around pods that are loading models, running defragmentation, or experiencing issues. No traffic reaches a pod that cannot serve it, eliminating the "healthy but not ready" failure mode entirely.
  • Session affinity -- Keeps multi-turn conversations on the same pod to maximize KV cache reuse. The second message in a conversation benefits from the cached context of the first, dramatically improving TTFT for conversational workloads.

The EPP transforms inference routing from a static load-balancing problem into a dynamic optimization problem. It makes thousands of informed decisions per second, each one considering the real-time state of the entire fleet.

The EPP essentially prevents many of the incidents described in this post by making smart, real-time decisions that account for the actual state of the system rather than relying on static configuration or naive load balancing.

What's Next

Production LLM inference is still a young discipline. The problems described here are actively being worked on by the llm-d community and by teams across the industry. Better scheduling algorithms, more efficient KV cache management, smarter memory allocation, improved observability tooling -- there is meaningful work to do on all fronts.

If your team has production inference experience, the llm-d community would benefit from it. The project is a CNCF Sandbox project, and contributions are welcome -- whether that is code, documentation, or sharing operational patterns from your own environment.
